# Parser

1. Про rate limit и баны - чтобы их избежать можно попробовать использовать несколько айпи или прокси, чтобы амазон думал что запросы идут  от разных пользователей или просто не слать ему большое кол-во запросов, а растягивать их насколько можем себе это позволить. Можно сильно не тормозить сразу, а ждать пока появятся логи, что парсер поймал капчу и тогда уже ставить паузы между запросами. Чем больше логов, тем больше должна быть задержка. Ещё, чтобы избежать банов, желательно хотя бы примерно иммитировать нормальные запросы, больше похожие на человеческие.

2. Про ошибки сети и таймауты - должны быть ретраи, поймали ошибку - подождали секунду, снова делаем запрос, если опять нет, то можно подождать 3 секунды, после 3 попыток сохранять в какую-нибудь табличку того что не получилось распарсить, чтобы не терять товар. С таймаутами - надо поставить примерное время сколько готовы ждать,  если за минуту нет ответа - можно отправить в очередь на доработку чтоб основной сервис не забивать или положить в ту же табличку с ошибками сети.

3. В случае изменения структуры html в данный момент вариант только переписать руками. Главное вовремя поймать ошибку и сделать алёрт. Может в скором времени это можно будет переложить на иишку, но щас это вряд-ли можно реализовать адекватно.

4. Логи желательно расставлять везде, во всех важных местах, логировать ссылки того что парсится и продумывать все моменты где что-то может серьёзно упасть и стараться как можно больше инфы дать по ошибке, опять же на самые важные логи слать алёрты что что-то идёт не так, особенно если таких логов полетит много.

По поводу выбора парсера - остановилась на cheerio, потому что он быстрее, остальные два нужны для кейсов когда cheerio не прошёл, тк он может только html парсить. Если бы делала сценарий на случай когда cheerio не смог распарсить, выбрала бы playwright, тк он масштабируется лучше и больше браузеров поддерживает.



Алгоритм работы парсера:

1. По ссылке заходит на страницу категории, сохраняет её айди, название и ссылку

2. Переходит к парсингу товаров, собирает asin, title, price, rating availability

3. После этого делает связку товара и категории

4. Если у товара есть продавец, связывает их, если нет, создаёт в бд

5. Парсит отзывы, собирает айди, рейтинг, лайки


Почему выбрала такую структуру

1. Снапшоты: нужны для статистики, без них никак, по продукту точно. Чтобы по товару таблица сильно не росла, сделано по триггеру, будь это продовый проект скорее всего пришлось бы даже ужесточить его, например если рейтинг был 4.8, а стал 4.9, то подождать ещё день, не делать снапшот сразу, цену можно было бы вообще убрать из условия. Смотря насколько мы можем позволить бд разрастись. Для продавца снапшот раз в 3 дня(не реализовано), но это просто задумка, возможно стоило бы делать его чаще/реже.

2. Seller: тк не у каждого товара есть продавец(как минимум у книг я не смогла его найти), пришлось делать модельку с типом seller/author. Весь амазон я не просмотрела, может где-то могут быть другие типы

3. Связь groupId у отзыва: на амазоне есть разные вариации товара(size/color), иногда заходя на продукт видишь ссылку на все отзывы сразу, а иногда только для этой конкретной вариации. Если убрать айди группы, он будет на каждую вариацию парсить одни и те же отзывы, если они вперемешку. 

4. Связь ProductCategory: иногда товар может быть в нескольких категориях сразу, поэтому желательно иметь такую таблицу с привязкой к каждой. Была идея сделать поле для продукта с массивом айдишек категорий, но это неправильно + не работали бы join и нормальный поиск по "все продукты в этой категории", поэтому решила что раз эта табличка не будет сильно большой, то можно её сделать
